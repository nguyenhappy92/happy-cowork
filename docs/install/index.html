Install · est. 60 seconds

Clone, symlink, cowork.

Happy Cowork installs as plain symlinks into ~/.cursor/. No build step, no package manager, no background service. Re-runs of install.sh are idempotent and safe.

Prerequisites

A typical developer machine already has these.

macOS or Linux

Windows works via WSL 2 (Ubuntu/Debian).

bash 4+

The installer uses set -euo pipefail and arrays. On macOS you can use the default /bin/bash 3.2 too — the script is compatible.

git 2.30+

For cloning and pulling updates.

Cursor

Any recent version. Launching Cursor at least once creates ~/.cursor/.

gh CLI optional

Used by create-pr, review-pr, triage-issues, release-notes. Run gh auth login once.

jq & direnv optional

JSON wrangling in scripts and per-repo env vars — nice to have, not required.

Quick install

Three commands. One minute.

  1. 1

    Clone the repo

    git clone https://github.com/nguyenhappy92/happy-cowork.git ~/code/happy-cowork

    Prefer SSH? Use git@github.com:nguyenhappy92/happy-cowork.git instead. You can also clone anywhere — ~/code/happy-cowork is a convention, not a requirement.

  2. 2

    Run the installer

    cd ~/code/happy-cowork && ./install.sh

    Idempotent — safe to re-run. Existing files at the target paths are backed up with a .bak suffix.

  3. 3

    Restart Cursor, then verify

    Open Cursor and ask the agent:

    › List the skills you have available.

    You should see create-pr, rebase, review-pr, daily-standup, triage-issues, and release-notes.

What the installer does

No magic. Just symlinks.

skills/* → ~/.cursor/skills/
rules/* → ~/.cursor/rules/
hooks/hooks.json → ~/.cursor/hooks/hooks.json
hooks/scripts → ~/.cursor/hooks/scripts
  • Because everything is a symlink, editing a file in the repo is instantly picked up by Cursor — no reinstall.
  • Existing symlinks at the target are replaced. Existing regular files/dirs are moved to <path>.bak before linking.
  • Scripts under hooks/scripts/ are chmod‑ed executable by the installer.
  • You can override the Cursor home with CURSOR_HOME=/custom/path ./install.sh.

Platform notes

macOS

Works with the bundled /bin/bash. If you use Homebrew bash (/opt/homebrew/bin/bash), that's fine too.

Linux

Any distro with bash 4+ and git. Tested on Ubuntu LTS and Debian stable.

Windows (WSL 2)

Install into your WSL home, not /mnt/c/…. Symlinks don't survive the DrvFs boundary reliably.

Updating

Since everything is symlinked, updating is just a git pull:

cd ~/code/happy-cowork && git pull --rebase

New skills or rules appear automatically because ~/.cursor/skills/ and ~/.cursor/rules/ point at the repo's folders. Restart Cursor if you added or changed hooks/hooks.json.

Uninstall

Remove the symlinks the installer created. Nothing else is touched — any .bak files stay where they are. 

cd ~/code/happy-cowork && \
find ~/.cursor/skills ~/.cursor/rules ~/.cursor/hooks -maxdepth 2 -type l \
  -lname "$(pwd)/*" -print -delete

Troubleshooting

Cursor doesn't see the skills.
Restart Cursor fully (quit, not just close the window). Confirm the symlinks exist: ls -la ~/.cursor/skills.
~/.cursor/ doesn't exist.
Launch Cursor once to create it, then re-run ./install.sh.
The installer says "Backing up existing … -> ….bak".
Expected. You had a regular file or directory at the target path; it's safe on disk next to the new symlink.
Permission denied on install.sh.
Run chmod +x install.sh once, then re-try.
I use zsh / fish — does that matter?
No. The script runs under bash regardless of your interactive shell.
Can I install into a custom Cursor home?
Yes: CURSOR_HOME=/some/path ./install.sh.
